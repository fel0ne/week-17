## 1. Обзор системы

Система предназначена для управления устройствами и автоматического асинхронного оповещения пользователей об изменении статуса их оборудования через различные каналы связи (`channel`: email, telegram, push).

Система состоит из двух основных микросервисов:
1. **Devices Service (`devices-svc`)** — отвечает за учет оборудования, генерацию событий (бизнес-логика).
2. **Notifications Service (`notifications-svc-s05`)** — принимает запросы на отправку уведомлений, валидирует каналы связи и осуществляет доставку.

## 2. Компоненты системы и их взаимодействие

Связь между внешним миром и сервисами организована через **API Gateway**, а межсервисное взаимодействие разделено на синхронное (gRPC) и асинхронное (Event-Driven через брокер сообщений).

```

[ Web / Mobile Client ]
           │ (REST / GraphQL)
           ▼
  ┌─────────────────┐
  │   API Gateway   │ (Prefix: /api)
  └────────┬────────┘
           │
           ├──────────────────────────────┐
           │ (REST / gRPC)                │ (/api/notifications)
           ▼                              ▼
┌────────────────────┐         ┌──────────────────────────┐
│    Devices Service │         │ Notifications Service    │
│    (devices-svc)   │         │ (notifications-svc-s05)  │
└──────────┬─────────┘         └──────────┬───────────────┘
           │                              │
           │ (Publish Event)              │ (Consume Event)
           ▼                              ▼
┌──────────────────────────────────────────────────────┐
│            Message Broker (RabbitMQ / Kafka)         │
└──────────────────────────────────────────────────────┘

```
### Спецификация интерфейсов `notifications-svc-s05` (Порт: 8151):

* **REST API:** доступен через шлюз по пути `/api/notifications`. Используется для получения истории отправленных уведомлений администратором или пользователем.
* **gRPC:** Пакет `notifications.v1`, сервис `NotificationsService`. Используется для высокопроизводительных синхронных внутренних запросов (например, когда `Devices Service` обязан жестко дождаться валидации канала перед регистрацией премиум-устройства).
* **GraphQL:** Тип `Notification`, query `notifications`, mutation `createNotification`. Используется фронтенд-клиентами для гибкой выборки истории уведомлений по конкретному каналу (`channel`) без оверхеда по трафику.

## 3. Базы данных и хранение данных

Каждый микросервис следует паттерну **Database-per-Service** для обеспечения слабой связности (Loose Coupling).

1.  **Devices DB (PostgreSQL):** Хранит информацию об устройствах (`id`, `name`, `serial`, `owner_id`).
2.  **Notifications DB (MongoDB или PostgreSQL):** Хранит историю нотификаций.
    * *Схема сущности `Notification`:*
        ```json
        {
          "id": "uuid",
          "user_id": "int",
          "title": "string",
          "body": "string",
          "channel": "string", // Поле из варианта (email/sms/tg)
          "status": "string",  // sent, failed, pending
          "created_at": "timestamp"
        }
        ```

## 4. Обеспечение надежности и слабой связности (Coupling & Fault Tolerance)

* **Асинхронность:** Сервис устройств не вызывает сервис уведомлений напрямую во время создания обычного девайса. Он бросает событие `DeviceCreatedEvent` в очередь сообщений. Если `notifications-svc-s05` упадет или уйдет на обслуживание, сообщения накопятся в брокере и обработаются позже. Бизнес-логика создания устройств не ломается.
* **Наблюдаемость (Observability):** Все логи пишутся в структурированном JSON-формате в `stdout`. В контейнере `notifications-container` настроен healthcheck эндпоинт (`/health`), опрашиваемый Kubernetes для контроля жизненного цикла пода (`liveness/readiness probes`).

---

## 5. Инфраструктура и CI/CD

* **Контейнеризация:** Все сервисы упакованы в легковесные Docker-образы (на базе `python:3.11-slim` или `alpine`).
* **Оркестрация (Локально):** Запуск всей инфраструктуры (Сервисы, БД, Брокер) осуществляется одной командой через `docker-compose.yml`.
* **Оркестрация (K8s):** Описаны манифесты для Deployment (`notifications-app`) и Service. Контейнер `notifications-container` ограничен по ресурсам (limits/requests CPU и Memory), чтобы не аффектить соседей по кластеру.
