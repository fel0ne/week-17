1. Какие критерии вы использовали при разделении монолита на микросервисы (или при проектировании сервисов)?
```
Разделение по функциональным ролям (бизнес-логике): Логика распределена на два независимых слоя. Первый — управление устройствами (devices-svc), второй — отправка оповещений через конкретные каналы связи (notifications-svc-s05), где обязательным параметром является тип доставки (channel).

Изоляция изменений: Логика добавления новых каналов уведомлений (например, интеграция с SMS-шлюзом или Telegram-ботом) изолирована внутри сервиса уведомлений и никак не требует пересборки или изменения кода сервиса устройств.
```
2. Почему вы выбрали именно этот стек технологий (язык, БД, протоколы)?
```
Язык и фреймворк: Python (FastAPI). Выбран из-за высокой скорости разработки веб-интерфейсов, встроенной валидации входящих данных (через схемы Pydantic) и поддержки асинхронности, что критично для сетевых запросов. Для GraphQL использовалась библиотека Strawberry.

База данных: На текущем этапе прототипа используется In-Memory хранилище (списки Python) внутри контейнера. Это позволяет мгновенно развернуть систему без накладных расходов на администрирование СУБД.

Протоколы: * REST API / GraphQL реализованы для интеграции с внешними клиентами. REST используется для стандартных запросов через API Gateway (маршрут /api/notifications), а GraphQL — для гибкого получения истории нотификаций без избыточности в данных.

gRPC (сервис NotificationsService) заложен в структуру проекта для легковесного бинарного взаимодействия между сервисами внутри контура.
```
3. Как ваши сервисы справляются с ошибками (retry, circuit breaker, graceful degradation)?
```
Graceful Degradation (Снижение функциональности): В коде devices-svc вызов внешнего сервиса уведомлений обернут в блок try-except. Если микросервис notifications-svc-s05 недоступен или падает с ошибкой, сервис устройств перехватывает исключение, логирует его, но не прерывает основной процесс. Пользователь успешно получает ответ о создании девайса, пусть и без отправки уведомления.

Сложные паттерны (Circuit Breaker, очереди с автоматическим Retry) на данном этапе проекта не реализовывались ради сохранения простоты архитектуры.
```
4. Как организован деплой и обновление вашей системы без простоя (Zero Downtime)?
```
Локально: Деплой осуществляется с помощью docker-compose.yml одной командой docker compose up -d. Обновление измененного Python-кода происходит без перезапуска самих контейнеров, так как директории проекта монтируются напрямую через volumes.

В Kubernetes (IaC): В подготовленном манифесте k8s/deployment.yaml для девелопмента notifications-app указан параметр replicas: 2. При обновлении версии образа Kubernetes использует встроенную стратегию Rolling Update — он поочередно запускает новые поды, проверяет их готовность и только потом гасит старые, что исключает появление ошибок 502 на шлюзе.
```
5. Что было самым сложным в интеграции всех компонентов вместе?
```
Самым сложным было совместить обработку трех разных интерфейсов (REST, gRPC, GraphQL) в рамках одного микросервиса на порту 8151 и настроить конфигурационный файл Nginx API Gateway так, чтобы он корректно проксировал только заданные в варианте пути (префикс /api/notifications), изолируя внутренние интерфейсы от внешнего доступа.

Также возникли сложности в CI/CD пайплайне при настройке автоматической проверки k8s-манифестов утилитой kubectl в изолированной среде GitHub Actions без подключения к реальному живому кластеру.
```
6. Что бы вы улучшили, если бы у вас был еще месяц?
```
Заменил бы In-Memory хранилище (списки в коде) на полноценную реляционную базу данных (например, PostgreSQL) с подключением ORM (SQLAlchemy) и миграций (Alembic).

Внедрил бы брокер сообщений (RabbitMQ), чтобы полностью перевести межсервисный обмен нотификациями на асинхронные рельсы и гарантировать доставку сообщений, даже если сервис уведомлений ушел в офлайн.

Дописал бы генерацию кода из .proto файла для полноценного запуска gRPC сервера параллельно с FastAPI.
```
